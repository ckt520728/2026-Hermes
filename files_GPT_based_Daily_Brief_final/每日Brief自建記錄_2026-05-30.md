# 從 Hermes Agent 到自建每日 Brief — 對話總結與踩坑記錄

> 日期：2026-05-30
> 主題：放棄 Hermes 的免費模型困境，改用 Google Apps Script 自建「每日研究 Brief」自動推播
> 成果：`daily-brief.gs`（抓 PubMed + arXiv + AI 新聞 RSS → GPT-5.5 整理 → 推 Telegram）

---

## 一、今天的決策路徑（為什麼最後選 Apps Script）

起點問題：Hermes Agent 搭配的免費大語言模型效果差，付費模型又另外收費。
既然已有付費的 Claude / GPT，想改用它們跑每日排程，達到 Hermes 的效果。

釐清過程中修正的幾個誤解：

1. **「用 Claude Cowork 排程 + 跑 GPT」不可行** — Cowork 底層是 Claude，無法選 GPT。
   且 Cowork 排程只在「電腦開著 + app 開著」時跑，對每天定時推播不可靠。
2. **「GPT-4.8」不存在** — GPT-4 系列已於 2026-02 退役；目前是 GPT-5 家族，
   API model 名稱為 `gpt-5.5`（或省錢用 `gpt-5.4`）。
3. **使用者實際用的是 Firebase，不是 Supabase**（偏好設定寫 Supabase，但帳號是 Firebase）。
   → 先前給的 Supabase 版 `index.ts` + `setup-cron.sql` 作廢。
4. **新 Firebase 專案是免費 Spark 方案** — Cloud Functions 需要 Blaze（付費、綁卡）才能對外連線。
   → 為了一個輕量任務不值得升級綁卡。

最終選擇：**Google Apps Script**
- 完全免費、不綁信用卡
- 全瀏覽器操作，不碰 CLI（最適合 CLI 不熟的情況）
- 內建觸發器（Trigger）= 雲端排程，跑在 Google 雲端，不靠自己電腦開著
- 任務輕（一天一次），不需要 Cloud Functions 這種重武器

## 三、今天踩過的坑（依發生順序）

### 坑 1：getUpdates 回 404 — Token 帶了角括號
- 現象：`api.telegram.org/bot<TOKEN>/getUpdates` 回 `{"ok":false,"error_code":404}`
- 原因：把範例的佔位符角括號 `< >` 一起打進網址了
- 解法：`bot` 後直接接 Token，不要任何括號／空格
- 附帶教訓：Token 一度連截圖外洩 → 立刻用 BotFather `/revoke` 作廢重發

### 坑 2：getUpdates 回 `result: []`（空的）
- 原因：還沒在「自己建的那個 bot」對話框發過話；getUpdates 只回最近、未讀的訊息
- 解法：先在 bot 對話框發一句話（必要時先按 START）→ 立刻重新整理 getUpdates
- 找 Chat ID：在回傳 JSON 裡找 `"chat":{"id":...}`（不是 update_id、不是 from.id）
- 本次 Chat ID：`8780377967`

### 坑 3：找不到 OpenAI 的 Billing 設定
- 真相：帳戶其實早已有餘額（Credit balance $7.29），billing 早就設好了
- 重要觀念：**OpenAI API 額度與 ChatGPT 訂閱是兩筆分開的錢**
- 新版介面：加值在右上「Add credits」、用量警示在「Manage spend alerts」

### 坑 4：Anthropic RSS 抓取失敗（HTTP 404）
- 原因：**Anthropic 官方根本沒有提供 RSS feed**；原本給的 anthropic.com/rss.xml 不存在
- 解法：改用社群維護的第三方 feed
  `https://raw.githubusercontent.com/taobojlen/anthropic-rss-feed/main/anthropic_news_rss.xml`
- 教訓：呼應 Hermes —「RSS 網址會變」，用 `testAINews()` 可單獨檢查哪家掛掉

### 坑 5：官方來源標題像「廢話」（標語式標題）
- 現象：Anthropic「Project Glasswing」、「Claude is a space to think」等，看標題猜不出內容
- 原因：公司官方發布頁的標題是品牌/標語型，真正內容在內文
- 解法 B：讓官方來源（desc:true）連 description/summary 一起抓
- 結果：**OpenAI 成功**（description 有真內容）；**Anthropic 無效**
  （第三方 feed 的 description = 標題本身，程式正確判斷重複並略過）
- 教訓：輸出品質的上限由「資料來源」決定，不是程式碼

### 坑 6：PubMed「rate limit exceeded」（今天最後遇到、已修）
- 現象：認知神經科學區塊抓不到文獻，顯示 API rate limit exceeded
- 原因：PubMed E-utilities 無 key 時每秒上限 3 次；兩次 PubMed 請求擠在同一秒
- 解法一（已寫入程式）：
  - fetchPubmed 內 esearch 與 efetch 之間 `Utilities.sleep(400)`
  - dailyBrief 兩次 PubMed 之間 `Utilities.sleep(600)`
- 解法二（選做）：申請免費 NCBI API key（每秒上限提到 10 次），
  設為指令碼屬性 `NCBI_API_KEY`，程式已支援「有就用、沒有也能跑」
- 教訓：同 Hermes —「免費資源有速率上限」，只是從每天 200 次變成每秒 3 次

## 四、設定備忘（三把 + 一把可選鑰匙）

存放位置：Apps Script → 專案設定 → 指令碼屬性

| 屬性名稱（須一字不差） | 內容 |
|---|---|
| `OPENAI_API_KEY` | OpenAI 的 sk-... 金鑰 |
| `TELEGRAM_BOT_TOKEN` | Telegram bot token（已 revoke 重發的版本）|
| `TELEGRAM_CHAT_ID` | `8780377967` |
| `NCBI_API_KEY` | （選用）NCBI API key，提高 PubMed 速率上限 |

驗證順序（逐段測，不要一次跑完）：
1. `testTelegram()` → 收到「✅ 測試」= Telegram 通
2. `testAINews()` → 看執行記錄，確認五家 RSS 哪些通
3. `dailyBrief()` → 收到完整四區塊 brief = 全通

## 六、部署與排程設定完成記錄（2026-05-31 更新）

### 坑 7：PubMed 頻率限制 (Rate Limit Access) 再度出現
- 原因：無 Key 情況下 PubMed API 每秒上限僅 3 次。
- 解決方法：
  - 申請 NCBI API Key 以提升每秒限制至 10 次。
  - 在 `daily-brief.gs` 新增 `setNcbiApiKeyHelper()` 快捷設定函式，讓使用者可直接在編輯器執行，安全且免手動點擊網頁選單的繁瑣操作。

### 觸發器排程設定完成（早上 7 點執行）
- 解決方法：
  - 在 `daily-brief.gs` 新增了 `setupDailyTrigger()` 函式，執行後自動在 Google 雲端建立每天早上 7:00 ~ 8:00 執行的觸發器。
  - 成功執行後會發送 Telegram 訊息確認「⏰ 觸發器設定成功！」。
