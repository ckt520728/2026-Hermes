## 二、踩過的坑：Notion API 刪除 blocks 導致資料永久遺失

### 錯誤經過

當我請 OWL 幫我重新整理一個演講筆記頁面時，OWL 的作法是：

1. 刪除頁面上所有現有的 40 個 blocks
2. 重新建立新的結構性 blocks（header、divider、section labels、summary）
3. 新的 Toggle blocks 全部是空的，原本的 AI 對話紀錄、研究筆記內容全部遺失

### 為什麼這是嚴重錯誤？

- **Notion API 沒有回收桶**：被刪除的 blocks 返回 404，無法透過 API 恢復
- **Toggle blocks 的 children 會一起消失**：Toggle 裡面的所有內容（AI 對話、研究筆記）在刪除父 block 時一併消失
- **link_to_page 的關聯被切斷**：原本連結到論文精讀頁面的連結全部遺失

### 補救方式

好在我有手動建立版本快照，所以可以從 Notion 的**頁面版本歷史**功能恢復。

### 教訓（已記錄為 OWL Memory）

> ⚠️ **Never delete existing blocks without preserving content first.**
> - Notion API has no recycle bin — deleted = permanent
> - Notion UI DOES have page version history for restoration
> - **Always INSERT new blocks between existing ones, never delete-then-rebuild**
> - Before restructuring: read ALL content → save backup → plan insertions → insert only

## 三、PARA 架構：第二大腦的骨架

### 什麼是 PARA？

PARA 是由 Tiago Forte 提出的知識管理架構，將所有內容分為四類：

| 類別 | 定義 | 我的範例 |
|------|------|---------|
| **🎯 Projects** | 有明確目標和截止日期的專案 | 博士論文口試準備 |
| **🔬 Areas** | 持續關注的研究領域 | AD研究、EEG分析、ML方法、AI工具 |
| **📚 Resources** | 參考資料、工具、模板 | 書籍/論文模板、教程、工具 |
| **🗄️ Archives** | 已完成或不再活躍的內容 | 不再活躍的舊頁面 |

### 為什麼需要 PARA？

沒有 PARA 結構的 Notion 資料庫，會變成**按時間順序堆積的流水帳**。所有新內容都平鋪在同一個層級，越積越多，難以檢索。

有了 PARA 之後：
- **Projects**：知道現在在做什麼
- **Areas**：知道長期關注什麼
- **Resources**：知道去哪裡找參考
- **Archives**：知道哪些已經完成

## 五、頁面結構設計原則

### Page-in-Page 卡片型結構

對於複雜的主題（如博士論文演講筆記），最佳結構是：

```
主頁面（演講大綱 / 目錄）
├── Toggle × N（各主題的筆記摘要）
├── link_to_page × N（連結到獨立的論文精讀子頁面）
└── child_page × N（技術筆記、方法細節）
```

**主頁面是「目錄」，Toggle 是「筆記摘要」，連結頁面是「論文精讀全文」**。

### 安全重構的正確姿勢

```
✅ 正確：INSERT new blocks between existing ones
✅ 正確：只修改 block 的 properties
✅ 正確：操作前完整讀取所有內容並備份
❌ 錯誤：Delete all → rebuild from scratch
❌ 錯誤：Delete toggle without preserving children
❌ 錯誤：Replace link_to_page with bookmark
```

### AI 對話紀錄的處理

AI 對話（ChatGPT、NotebookLM、Gemini）是學習過程中的重要產出，應該：
- 保留在 Toggle 中（收合式，不佔空間）
- 與正式筆記內容分離（放在獨立的區域）
- 標注日期和使用的 AI 工具
